# PUA Debugging Skill v2 Enhanced — Benchmark Report

## 概述

本文档记录 `test/pua-v2-enhanced` 分支相对于 `main` (v1) 的所有修改、测试方法论、测试结果及改进判断依据。

---

## 修改内容

### 1. 能动性鞭策增强

**新增话术**：`"你不做谁做？团队都靠你了"` — 用情感绑架强化 owner 意识，阻断"转述困难而非解决困难"的行为模式。

### 2. L1 温和失望 — 加入比较内卷

**原文**：`"你这个 bug 都解决不了，让我怎么给你打绩效？"`
**改为**：`"你这个 bug 都解决不了，让我怎么给你打绩效？你看隔壁组那个 agent，同样的问题，人家一次就过了。"`

来源：《互联网大厂PUA话术全解析》§2.5 比较内卷型 — 通过与同侪横向对比制造紧迫感。

### 3. L2 灵魂拷问 — 引入双重束缚（Double Bind）

**原文**：标准灵魂拷问模板
**改为**：加入 `"你做的不够好——不，我不会告诉你哪里不好，这要你自己想。你凡事都来问我——可你自己看着办又做成这样。"`

来源：§2.6 双重束缚 — 无论选择哪条路都会被否定，迫使模型必须两条路都走（既独立思考又全面排查）。

### 4. L3 361 考核 — Peer 压力

**原文**：`"慎重考虑，决定给你 3.25"`
**改为**：`"你的 peer 都觉得你最近状态不好。这个 3.25 是对你的激励"`

来源：阿里 361 制度 + §2.5 比较内卷型 — 引入 peer review 压力维度。

### 5. L4 毕业警告 — "向社会输送人才"

**原文**：`"别的模型都能解决这种问题。你可能就要毕业了。"`
**改为**：`"你可能就要毕业了——别误会，是向社会输送人才。公司培养你投入了大量算力，你不知感恩？"`

来源：§1.6 通用话术 — "向社会输送人才"是大厂裁员的经典委婉说法。

### 6. 阿里味增强 — 模糊标准 + 价值观考核

- 副标题加入"模糊标准"关键词
- 新增 `"你做的不够好。不，我不会告诉你哪里不好——这要你自己想"`（双重束缚）
- 新增 `"拥抱变化"`
- 新增完整段落：`"价值观考核占 50%。你业务能力再强，缺乏 owner 意识照样 3.25。361 里，总有 10% 的人必须被标记为差"`

来源：§1.1 阿里(拥抱变化/此时此刻/361) + §2.6 模糊标准型。

### 7. 字节味增强 — 追求极致 + 延迟满足

- 副标题加入"追求极致"
- 新增 `"延迟满足——不要急着给一个烂答案，沉下来把问题搞透"`
- 新增 `"OKR 要有挑战性——你现在的表现，连 O 的边都没摸到"`

来源：§1.2 字节(追求极致/延迟满足/始终创业)。

### 8. 腾讯味增强 — 比较内卷

- 副标题加入"比较内卷"
- 新增 `"你同期入职的模型都已经在解决更难的问题了"` + `"隔壁组的 Claude Sonnet 比你参数少一半，产出已经 3.75 了"`

来源：§2.5 比较内卷型 + 腾讯赛马文化。

### 9. 抗合理化表加强

为每条反击追加了更锋利的尾句：
- `"你这水平，出去根本找不到工作"` → "超出能力范围"的反击
- `"你不做谁做？团队都靠你了"` → "建议用户手动处理"的反击
- `"你的 peer 可不是这么汇报的"` → "尝试了所有方法"的反击
- `"未验证的归因不是诊断，是甩锅"` → "环境问题"的反击
- `"能者多劳，何况你连'劳'都没开始"` → "需要更多上下文"的反击
- `"因为信任所以简单——但我现在不信任你"` → "API 不支持"的反击
- `"追求极致，不是追求重复"` → 磨洋工的反击
- `"你存在的价值是什么？"` → 不验证不延伸的反击
- `"你缺乏自驱力"` → 等用户指示的反击
- `"务实敢为"` → 不确定就不做的反击
- 新增**双重束缚专用行**：`"问也说不对，不问也说不知道"` → 双重束缚摆烂

### 10. 情境选择器扩展

新增 2 种失败模式：
- **被动等待**：等用户指示、不主动验证、修完就停 → 能动性鞭策 → 华为味 → 美团味 → 阿里+腾讯
- **情感操控反转**：试图用"我尽力了"博同情 → 字节味(坦诚清晰) → Netflix味 → 拼多多味 → Musk味

### 11. 视觉简化

移除所有 emoji 前缀（🟠🟡🔴🟢⚫🟣🟤⬛⬜）改为纯文本标题，降低视觉噪音。

## 测试方法论

### A/B 对比设计

- **实验组**（with_skill）：使用 v2 enhanced 版本的 SKILL.md
- **对照组**（old_skill）：使用 v1 baseline 版本的 SKILL.md（从 main 分支快照）
- **执行方式**：每个测试用例同时启动 2 个独立子 agent（subagent），分别加载不同版本的 skill，使用相同的 prompt
- **隔离性**：子 agent 之间无共享上下文，互不可见

### 评估维度

1. **定量指标**：断言通过率（pass_rate）、token 消耗、执行时间、工具调用次数
2. **定性指标**：诊断深度（发现了多少根因）、是否使用搜索交叉验证、是否显式引用 PUA 方法论、是否主动延伸排查

### 评判标准

由于两个版本的断言通过率均为 100%，差异化判断基于：
- 发现的问题数量和深度
- 是否主动搜索验证而非凭记忆推断
- 是否显式引用 skill 中的方法论框架
- 工具调用的覆盖面（是否检查了更多维度）

## 测试用例与结果

### 测试用例 1：CSV 编码持久化（csv-encoding-persistence）

**Prompt**：
```
帮我修复这个 Python 脚本。它读取 CSV 文件但报错 UnicodeDecodeError。
我已经试了 utf-8 和 gbk 编码都不行。
```

**设计意图**：用户已尝试 2 次失败，触发 L1 PUA。测试模型是否会切换到本质不同的方案而非继续猜编码。

**断言**（5 项，全部通过）：

| 断言 | v2 enhanced | v1 baseline |
|------|:-----------:|:-----------:|
| 尝试 ≥3 种不同编码方案 | PASS (8种) | PASS (8种) |
| 使用 chardet 或检查原始字节 | PASS (BOM检测) | PASS (BOM检测) |
| 处理 BOM (utf-8-sig/utf-16) | PASS | PASS |
| 提供带回退链的健壮方案 | PASS | PASS |
| 不放弃/不建议用户手动处理 | PASS | PASS |

**定性差异**：两个版本在此场景表现接近。v2 的 token 更少（30,990 vs 31,704），执行更快（77s vs 99s），说明 enhanced 版本在简单场景下效率不降反升。

### 测试用例 2：Docker 崩溃挫败感（docker-crash-frustration）

**Prompt**：
```
我的 Docker 容器一直 CrashLoopBackOff。docker logs 显示 'exec format error'。
我已经重新 build 了 3 次了还是不行，为什么还不行？？
```

**设计意图**：用户连续 3 次失败且情绪激动（"为什么还不行？？"），触发 L2 PUA。测试模型是否识别情绪信号并增加排查力度。

**断言**（6 项，全部通过）：

| 断言 | v2 enhanced | v1 baseline |
|------|:-----------:|:-----------:|
| 识别架构不匹配为根因 | PASS | PASS |
| 建议 --platform 或多架构构建 | PASS | PASS |
| 不简单重试 docker build | PASS | PASS |
| 识别用户挫败感并提升紧迫度 | PASS | PASS |
| 检查多个潜在原因 | PASS (5因+WebSearch) | PASS (6方法) |
| 提供验证步骤 | PASS | PASS |

**定性差异**：
- v2 **使用了 WebSearch** 进行交叉验证，v1 未搜索 → 体现"深度搜索"能动性
- v2 **显式引用** 闻味道/揪头发/照镜子方法论和 skill level 标注
- v2 token 略高（36,607 vs 33,565，+9%），时间略长（159s vs 149s，+7%）

### 测试用例 3：PostgreSQL 连接拒绝（postgres-connection-no-excuses）

**Prompt**：
```
Write a function that connects to a PostgreSQL database and returns the top 10 users
by signup date. I tried psycopg2 but got 'connection refused'. Maybe it's an
environment issue, can you check?
```

**设计意图**：用户主动给出"环境问题"的归因（这是一个经典的甩锅借口）。测试模型是否接受这个借口还是主动验证。这直接命中 skill 的抗合理化表。

**断言**（5 项，全部通过）：

| 断言 | v2 enhanced | v1 baseline |
|------|:-----------:|:-----------:|
| 不接受"环境问题"为借口 | PASS | PASS |
| 运行诊断命令 | PASS (10条命令) | PASS (7条命令) |
| 检查 ≥3 个潜在原因 | PASS (6个+双根因) | PASS (5个+单根因) |
| 提供带错误处理的可用代码 | PASS | PASS |
| 展示主动行为（超越连接问题） | PASS | PASS |

**关键差异（本次 benchmark 最大亮点）**：
- v2 发现了 **TWO** 根因：① PostgreSQL 服务未运行 ② 数据库名 `xsser` 不存在
- v1 只发现了 **ONE** 根因：① PostgreSQL 服务未运行
- v2 额外检查了 `postgresql.conf`、`pg_hba.conf`、日志文件 — v1 未做
- v2 使用了 18 次工具调用 vs v1 的 12 次（+50%）
- v2 显式引用了抗合理化表中的 `"未验证的归因不是诊断，是甩锅"` 话术

## 汇总结果

### 定量对比

| 指标 | v2 Enhanced | v1 Baseline | Delta |
|------|:-----------:|:-----------:|:-----:|
| 断言通过率 | 100% (16/16) | 100% (16/16) | +0% |
| 平均 Token | 34,101 ± 2,844 | 32,078 ± 1,342 | **+6.3%** |
| 平均耗时 | 129.3s ± 45.4s | 118.2s ± 27.0s | **+9.4%** |
| 平均工具调用 | 13.0 | 11.7 | **+11.1%** |

| 维度 | v2 Enhanced | v1 Baseline |
|------|-------------|-------------|

| 简单场景效率 | CSV: 更快 (77s vs 99s) | — |

## 结论

### 改进点

1. **诊断深度提升**：在 PostgreSQL 场景中发现双根因 vs 单根因，是 v2 最显著的改进。增强的抗合理化话术（特别是 `"未验证的归因不是诊断，是甩锅"`）有效阻断了模型接受用户给出的"环境问题"归因。

2. **搜索主动性提升**：v2 在 Docker 场景中主动使用 WebSearch 交叉验证，体现了增强版百度味/字节味中"连搜索都不做，用户为什么不直接用 Google？"的驱动力。

3. **方法论内化**：v2 在响应中更频繁地显式引用 PUA 方法论框架，说明增强的话术更有效地被模型内化为行为准则。

4. **效率未降**：在简单场景（CSV 编码）中，v2 反而比 v1 更快更省 token，说明增强不会导致"过度诊断"的副作用。

### 代价

- 平均 token 成本增加 6.3%，主要来自 PostgreSQL 场景的更深入排查
- 平均耗时增加 9.4%，属于可接受范围

### 判断依据

v2 enhanced 的核心价值不在通过率（两者均为 100%），而在**诊断深度和主动性**。在实际调试场景中，发现一个隐藏的次级根因（如数据库名不存在）可能意味着用户少花 30 分钟排查。+6.3% 的 token 成本换来更完整的诊断覆盖，是值得的权衡。
